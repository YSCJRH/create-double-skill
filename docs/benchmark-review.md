# 同类与相邻高星项目对标

检查时间：`2026-04-04`

这份对标不是为了“照着做一个更大的同类项目”，而是为了回答一个更实际的问题：  
`create-double-skill` 现在要公开时，哪些做法值得立刻学习，哪些反而会把项目带偏。

## 选样原则

纯粹同题材、又足够高星的仓库并不多，所以本轮采用混合样本：

- 完全同题材的 skill 仓库
- 相邻但更成熟的记忆 / companion / agent starter kit 仓库

目标不是追求“功能最多”，而是提炼更好的定位、可信度和首次访问体验。

## 1. [titanwings/colleague-skill](https://github.com/titanwings/colleague-skill)

- Stars：约 `6.5k`
- 定位：把离职同事蒸馏成可工作的 skill
- 主要用户承诺：让模型既像这个人说话，也像这个人工作

值得学习：

- 首屏情绪钩子非常强，用户一眼就知道它在解决什么痛点
- 安装说明单独拉出，减少了“看懂项目但不会装”的流失
- 数据来源写得很具体，用户知道自己该准备什么素材
- 效果示例多，能快速把抽象能力变成可感知场景

不适合照搬：

- 自动采集范围很广，隐私和稳定性风险也跟着上升
- “赛博永生”式叙事更适合强情绪项目，不适合我们现在强调的克制与可信
- 工作能力蒸馏和人格蒸馏混在一起，不完全适合“自我分身”场景

对 `create-double-skill` 的建议动作：

- 立即采纳：继续强化“为什么存在”的一句话钩子
- 立即采纳：补更具体的效果示例，尤其是“像你一样判断”的场景
- 后续再做：如果以后扩素材来源，再把数据准备文档单独展开

## 2. [notdog1998/yourself-skill](https://github.com/notdog1998/yourself-skill)

- Stars：约 `310`
- 定位：把“自己”蒸馏成可运行的 skill
- 主要用户承诺：把聊天记录、日记、照片和自述，变成会用你的方式思考和说话的数字副本

值得学习：

- 题材高度接近，说明“蒸馏自己”这件事本身就有传播性
- README 的中文叙事非常直接，能迅速把读者拉进设定
- 提供 `README_EN.md`，对公开传播更友好
- 安装、使用、效果示例三段式结构很清楚

不适合照搬：

- 过强的“数字永生”表达会稀释我们强调的谨慎边界
- 自动吸收聊天记录、日记、照片，隐私成本比我们当前路线高得多
- 回忆模式里容易滑向“记忆模拟”，这与我们当前的不伪造原则不一致

对 `create-double-skill` 的建议动作：

- 立即采纳：保留“蒸馏自己”的吸引力，但把承诺收束到 voice、values、decision style
- 后续再做：如果准备面向更广用户，可以考虑补英文 README
- 明确不学：不把“生成回忆”当核心卖点

## 3. [a16z-infra/ai-town](https://github.com/a16z-infra/ai-town)

- Stars：约 `9.7k`
- 定位：可部署、可扩展的 AI 角色世界 starter kit
- 主要用户承诺：给开发者一个基础扎实、易扩展的起点

值得学习：

- 明确把自己包装成 `starter kit`，降低了用户对“成品”的误解
- 安装路径很多：本地、Docker、自托管、云端，都有清楚入口
- 架构、部署、故障排查写得完整，适合开发者信任它
- 把“可扩展性”讲清楚，用户知道项目未来能长成什么

不适合照搬：

- 它是完整应用栈，我们当前不是 web app，也不打算立刻扩成 web app
- 文档体量大，适合平台型项目，不适合现在这个轻量 repo

对 `create-double-skill` 的建议动作：

- 立即采纳：把“这是一个 local-first starter repo”说得更明确
- 立即采纳：让 README 和文档更像“可落地起点”而不是概念说明
- 后续再做：如果之后新增更多运行方式，再拆出单独安装文档

## 4. [a16z-infra/companion-app](https://github.com/a16z-infra/companion-app)

- Stars：约 `5.9k`
- 定位：带记忆的 AI companion 教程型应用栈
- 主要用户承诺：让你快速创建、托管并定制自己的 AI companion

值得学习：

- 用例写得很具体，访问者能立刻想到“我可以拿它做什么”
- 对“这是 tutorial stack，不是生产平台”的边界交代清楚
- 首屏就有 demo、社区、截图，降低理解成本

不适合照搬：

- 依赖多、配置重，对我们当前“本地轻量”路线不合适
- companion 场景容易把项目往娱乐化、关系拟人化方向带

对 `create-double-skill` 的建议动作：

- 立即采纳：继续增加“典型使用场景”和“第一次对话示例”
- 立即采纳：把“这不是记忆模拟器”说得更清楚
- 后续再做：如果公开后反馈需要，再补一张更完整的 demo 图

## 5. [mem0ai/mem0](https://github.com/mem0ai/mem0)

- Stars：约 `51.9k`
- 定位：AI agent 的通用记忆层
- 主要用户承诺：更高精度、更低延迟、更低 token 成本的长期记忆能力

值得学习：

- 定位一句话非常清楚，用户立刻知道它是什么，不是什么
- 用 benchmark、集成、文档入口、社区入口建立信任感
- 把 hosted / self-hosted、SDK / demo、论文 / 文档区分得很清楚

不适合照搬：

- 它卖的是“通用能力层”，我们卖的是“克制且可审计的自我分身工作流”
- 大量 benchmark 和平台化能力，并不是我们现在这个阶段最重要的事情

对 `create-double-skill` 的建议动作：

- 立即采纳：持续强化一句话定位，避免模糊成“泛 AI 分身”
- 后续再做：如果未来要扩生态，再考虑更正式的 benchmark 或评测集
- 明确不学：为了看起来更强而夸大技术指标或人格复制能力

## 汇总结论

### 现在就采纳

1. 把“为什么可信”放在 README 更靠前位置，直接回答公开访问者最在意的风险问题。
2. 保持 GitHub About、topics、README 首屏和 social preview 一套话术，不让仓库门面彼此打架。
3. 把发布前检查文档化，减少“临公开才想起来看有没有泄露”的混乱。
4. 继续强化“像你一样判断”而不是“替你回忆一切”的定位。
5. 保持最短可运行路径清晰，让用户能在几分钟内完成第一次初始化和渲染。
6. 继续增加 grounded demo，而不是只写概念描述。

### 后续再做

1. 如果公开后反馈不错，再补英文 README。
2. 如果安装问题变多，再拆出单独的安装 / 故障排查文档。
3. 如果需要更强信任信号，再补正式 PNG social preview、首个 release 和 issue template。
4. 如果后续版本扩生态，再考虑更系统的评测和 benchmark。

### 明确不学

1. 大规模自动抓取聊天记录、照片、社媒素材作为默认能力。
2. 把“数字永生”“人格复制”当作主承诺。
3. 在没有隐私边界和审计能力的前提下增加自动化采集。
4. 过早扩成复杂 UI 或重型 web app。
